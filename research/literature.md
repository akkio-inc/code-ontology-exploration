# Literature Review: Human vs. Agent Code Structural Signatures

> Last updated: 2026-04-09

This document catalogs research sources informing the Code Ontology project — papers, blog posts, and tools relevant to detecting, classifying, and measuring structural differences between human-authored and AI-agent-authored code.

---

## Academic Papers

### 1. Fingerprinting AI Coding Agents on GitHub

- **Authors:** Taher A. Ghaleb
- **Venue:** MSR '26 (23rd International Conference on Mining Software Repositories)
- **Date:** January 2026
- **arXiv:** [2601.17406](https://arxiv.org/abs/2601.17406)

**Summary:** Analyzed 33,580 PRs from five major AI coding agents (OpenAI Codex, GitHub Copilot, Devin, Cursor, Claude Code) to identify behavioral fingerprints. Used 41 features spanning commit messages, PR structure, and code characteristics.

**Key findings:**
- Achieved **97.2% F1-score** in multi-class agent identification
- Top discriminating feature for Codex: **multiline commit message ratio** (67.5% feature importance)
- Top discriminating feature for Claude Code: **conditional statement density** (27.2% importance)
- Commit message structure was the single most powerful feature category overall

**What we adopted:**
- Multiline commit ratio metric (implemented in `topology.py`)
- Conventional commit ratio metric
- Co-author tag detection
- Footer tag detection
- Average message line count
- These were the "easy metrics" — computationally cheap and highly discriminating

### 2. The Hidden DNA of LLM-Generated JavaScript: Structural Patterns Enable High-Accuracy Authorship Attribution

- **Authors:** Norbert Tihanyi, Bilel Cherif, Richard A. Dubniczky, Mohamed Amine Ferrag, Tamás Bisztray
- **Date:** October 2025 (revised November 2025)
- **arXiv:** [2510.10493](https://arxiv.org/abs/2510.10493)
- **Category:** cs.CR, cs.LG

**Summary:** Investigated whether JavaScript code from different LLMs exhibits unique stylistic signatures. Built the LLM-NodeJS dataset: 50,000 Node.js programs from 20 LLMs, yielding 250,000 samples in three representations (original, JSIR, AST).

**Key findings:**
- **95.8% accuracy** on 5-class, **88.5%** on 20-class attribution
- Classifiers capture "deeper stylistic regularities in program **dataflow and structure**" — not surface features
- Attribution survived code mangling, comment removal, and heavy transformations
- Custom CodeT5-JSA model (770M parameters) performed best
- **Dataflow structure is the strongest AI code signal** — more robust than surface-level features

**What we could adopt (medium/hard difficulty):**
- AST-level structural analysis (function shapes, nesting patterns)
- Dataflow graph metrics
- Code representation transformations (original → IR → AST) for robust comparison

### 3. Automatic Detection of LLM-Generated Code: A Comparative Case Study

- **Authors:** Musfiqur Rahman, SayedHassan Khatoonabadi, Ahmad Abdellatif, Emad Shihab
- **Date:** September 2024 (revised December 2025)
- **arXiv:** [2409.01382](https://arxiv.org/abs/2409.01382)

**Summary:** Analyzed 14,485 Python functions and 11,913 classes generated by four LLMs (GPT-3.5, Claude 3 Haiku, Claude Haiku 4.5, GPT-OSS) using interpretable software metrics and CatBoost classifiers.

**Key findings:**
- **Comment-to-Code Ratio** is the "sole universal discriminator" across all models
- GPT-3.5 highly detectable (AUC-ROC 0.96); modern models harder (0.68–0.80)
- **Granularity effects dominate model differences by 8.6x** — class-level detection is very different from function-level
- Negligible feature overlap between granularity levels
- Modern models are more detectable at class level; older models at function level

**What we could adopt (medium difficulty):**
- Comment-to-code ratio (needs AST or line-level parsing)
- Function-level vs class-level granularity analysis
- Per-model detection characteristics

### 4. git2net — Mining Time-Stamped Co-Editing Networks from Large git Repositories

- **Authors:** Christoph Gote, Ingo Scholtes, Frank Schweitzer
- **Venue:** MSR '19 (16th International Conference on Mining Software Repositories)
- **Date:** March 2019
- **arXiv:** [1903.10180](https://arxiv.org/abs/1903.10180)

**Summary:** Introduced git2net, a Python tool for extracting fine-grained co-editing networks from git repositories. Uses text mining to analyze detailed history of textual modifications within files, constructing directed, weighted, time-stamped networks where links signify one developer editing code originally written by another.

**Key findings:**
- Co-editing networks capture collaboration patterns invisible at file/module level
- Networks are directed (who edits whose code), weighted (how much), and temporal
- Applied to both open-source and commercial projects
- Provides "high-resolution data on human collaboration patterns"

**What we could adopt (hard difficulty):**
- Co-editing network construction (who overwrites whose code — do agents overwrite human code more?)
- Temporal collaboration patterns (does agent code get edited by humans more quickly?)
- Network topology differences between human and agent collaboration graphs

## Industry Blog Posts

### 5. CodeRabbit: "2025 was the year of AI speed. 2026 will be the year of AI quality"

- **Source:** [CodeRabbit Blog](https://www.coderabbit.ai/blog/2025-was-the-year-of-ai-speed-2026-will-be-the-year-of-ai-quality)
- **Date:** ~January 2026

**Summary:** Analysis of AI code quality based on CodeRabbit's code review data across many repositories.

**Key findings:**
- AI-generated code has **1.7x more issues and bugs** than human-written code
- AI code contains **up to 75% more logic and correctness issues** in areas likely to cause downstream incidents
- 2025 saw "unprecedented number of incidents" with postmortems pointing to AI-generated code
- Issues: subtle logic errors, configuration oversights, design misunderstandings, late-stage defects
- Reviewing AI code is "more cognitively demanding than writing it from scratch"
- Subtle errors are "easy to miss in large, machine-generated diffs"

**Relevance to our project:**
- Validates that measurable quality differences exist between human and AI code
- Suggests specific metric categories: logic errors, configuration issues, defect density
- The "cognitive demand" finding could correlate with structural complexity metrics

### 6. CodeRabbit: "Misalignment: The hidden cost of AI coding agents isn't from AI at all"

- **Source:** [CodeRabbit Blog](https://www.coderabbit.ai/blog/the-hidden-cost-of-ai-coding-agents-isnt-from-ai-at-all)
- **Date:** ~2026

**Summary:** Argues the real cost of AI coding isn't model quality but misalignment between intent and execution.

**Key findings:**
- "The code can be perfectly valid, logically sound, and honestly impressive… and still be completely wrong"
- AI detection accuracy estimated at **80–90%** depending on whether explicit attribution markers are present
- Teams experience compounding costs through repeated prompt refinement cycles
- Faster execution means unclear requirements transform into massive, hard-to-review PRs

**Relevance to our project:**
- PR size / diff size as a metric — agent PRs may be systematically larger
- Review cycle length as a signal

### 7. CodeRabbit: Framework for Evaluating AI Code Review Tools

- **Source:** [CodeRabbit Blog](https://www.coderabbit.ai/blog/framework-for-evaluating-ai-code-review-tools)
- **Date:** January 9, 2026

**Summary:** Proposes seven metric categories for evaluating AI code review: architectural, adoption, engagement, impact, quality/security, governance, and developer sentiment.

**Key findings:**
- "Benchmarks don't measure 'quality' in the abstract. They measure whatever the benchmark designer chose to emphasize"
- Recommends **acceptance rates** (% of AI comments → code changes) as primary quality signal
- Running multiple AI tools simultaneously creates "interference effects"

**Relevance to our project:**
- Reminds us that our metrics measure what we choose to measure — need to be explicit about what each metric captures
- Acceptance rate concept could inform how we weight our findings

## Tools & Detection Systems

### 8. Coderbuds: Open-Source AI Code Detection YAML Rules

- **Source:** [Coderbuds Blog](https://coderbuds.com/blog/open-source-ai-code-detection-yaml-rules)
- **Date:** December 31, 2025

**Summary:** Open-sourced YAML-based detection rules for identifying AI-generated pull requests. Practical system used in production.

**Detection patterns by tool:**

| Tool | Method | Confidence |
|------|--------|-----------|
| Claude Code | Footer (`[Claude Code](https://claude.com/...)`) + co-author attribution | 100% |
| GitHub Copilot | Bot commits (`github-copilot[bot]`) | 100% |
| Cursor | Footer references + branch patterns (`cursor-` prefix) | 90% |
| Aider | Commit attribution patterns | 85% |
| WindSurf | Footer mentions | 95% |
| OpenAI Codex | Branch patterns (`codex-` prefix) | 60% |

**Detection categories:**
1. **Commit footers** — tool signatures in PR descriptions / commit messages
2. **Bot authors** — bot email addresses in commit metadata
3. **Co-author attribution** — `Co-Authored-By:` lines naming AI tools
4. **Branch naming patterns** — tool-specific prefixes (lower confidence)
5. **PR labels & HTML comments** — tool-specific markers like `<!-- Cursor -->`

**What we adopted:**
- Directly informed our 3-layer author classification system in `author_classifier.py`
- Email patterns, commit message footers, and bot heuristic all derive from these categories

### 9. Coderbuds: PR-Based AI Tool Detection Approach

- **Source:** [Coderbuds Blog](https://coderbuds.com/blog/tracking-ai-coding-tools-pull-request-based-approach)
- **Date:** December 28, 2025

**Summary:** Explains why they chose PR-level detection over API integrations with AI tools.

**Key findings:**
- **Explicit attribution** covers ~75% of AI-generated code (bot commits, footers, co-author tags)
- Remaining 25% requires **pattern analysis**: commit patterns, message quality, code structure
- "Unusually verbose commit messages with perfect grammar" and "highly structured PR descriptions" are AI indicators
- Overall detection accuracy: 80–90%
- Three core metrics: AI adoption rate, AI-assisted PR rate, tool breakdown

**Relevance to our project:**
- Validates our multi-layer detection approach
- The 75/25 explicit/pattern split matches our experience (email detection catches some, message scanning catches most of the rest)
- "Verbose with perfect grammar" is a measurable signal we haven't implemented yet

## Research Gaps & Opportunities

Based on the literature review, these areas are underexplored:

1. **Longitudinal structural evolution** — How does code structure change over time as AI adoption increases within a single repo? No papers found measuring this.

2. **Co-change topology differences** — Do agent-authored files form different co-change network shapes than human-authored files? git2net provides the tooling but nobody has applied it to this question.

3. **Burstiness and temporal patterns by author type** — We appear to be novel in measuring inter-event time distributions split by human vs agent authorship.

4. **Cross-repo structural comparison** — Most papers study individual repos or synthetic benchmarks. Our multi-specimen comparison across the human→agent spectrum appears novel.

5. **Commit message structure as a first-class metric** — The MSR '26 paper validates this but doesn't provide the feature as a standalone analysis tool. Our implementation makes this accessible.
